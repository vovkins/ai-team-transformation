# AI-Augmented Team Transformation Guide

Руководство по внедрению AI-агентов в работу команды разработки цифрового продукта.

## Контекст

Команда из 10 человек (разработчики, дизайнеры, системные аналитики, архитектор, продакт-оунер, QA) работает над цифровым продуктом. Команда привыкла к традиционному SDLC без AI-инструментов. У команды нет опыта использования AI-агентов.

## Цель

Кратно ускорить рабочий процесс команды с помощью AI-агентов, не разрушив то, что уже работает. Дать каждому члену команды персонального AI-агента, который усилит его экспертизу и повысит продуктивность.

## Ключевой принцип

**Агент не заменяет экспертизу — агент расширяет и усиливает её.**

Каждый член команды становится «tech lead» для своего AI-агента: направляет, проверяет, принимает решения. Агент берёт рутину и ускоряет execution.

## 1. Роли и зоны ответственности

Каждая роль имеет **двойную ответственность:**

1. **Развивать свой инструментарий** — настраивать, обучать, адаптировать один или несколько AI-агентов (навыков) под задачи своей специализации
2. **Производить качественный AI-friendly артефакт** — результат работы должен быть в формате, удобном для агентов других ролей, и пополнять общий контекст проекта

### Продакт-оунер (PO)

| Что делает | Как использует агента | Артефакт на выход |
|---|---|---|
| Продуктовое исследование | Конкурентный анализ, гипотезы, исследование рынка | Product Research Report |
| Формулирует vision проекта | Структурирование идей, дополнение требований | PRD |
| Декомпозирует на фичи и user stories | Разбивка на stories, поиск пропусков | Product Backlog |
| Приоритизирует бэклог | Оценка impact, поиск зависимостей | Приоритизированный Backlog |

### Архитектор

| Что делает | Как использует агента | Артефакт на выход |
|---|---|---|
| Проектирует архитектуру | Оценка стека, архитектурные паттерны | Architecture Document |
| Формирует стандарты (CONVENTIONS.md) | Паттерны, анти-паттерны, best practices | Architectural Guidelines |
| Определяет модульные границы | Выделение модулей, интерфейсы | Module Boundaries |
| Контролирует архитектурную целостность | Проверка PR на соответствие стандартам | Architecture Review |

### Системный аналитик

| Что делает | Как использует агента | Артефакт на выход |
|---|---|---|
| Декомпозирует stories в спецификации | Проверка полноты, edge cases, валидация | Task Specification |
| Описывает API, контракты, бизнес-логику | Генерация схем, проверка консистентности | API Specification |

### UI/UX Дизайнер

| Что делает | Как использует агента | Артефакт на выход |
|---|---|---|
| Проектирует интерфейсы | Конвертация макетов в код, дизайн-система | Design System + UI Specs |
| Описывает interaction patterns | Accessibility, варианты для edge cases | Component Library Spec |

### Разработчик

| Что делает | Как использует агента | Артефакт на выход |
|---|---|---|
| Реализует задачи по спецификации | Пишет код, тесты, рефакторит | Source Code + Unit Tests |
| Code review | Первый проход (lint, стиль, баги) | Review Comments |

### QA-инженер

| Что делает | Как использует агента | Артефакт на выход |
|---|---|---|
| Проектирует тест-стратегию | Генерация тест-кейсов | Test Plan + Test Cases |
| Пишет автотесты | Генерация автотестов, анализ coverage | Automated Tests |

## 2. SDLC: шаги и артефакты

Каждый шаг создаёт чёткий артефакт в AI-friendly формате. Артефакт становится частью контекста проекта.

### Формат артефактов

Все артефакты строятся по принципам, удобным и для людей, и для AI-агентов:

- **Структурированный markdown** — человек читает как документ, агент парсит по секциям
- **Связи через идентификаторы** — `US-42`, `TS-15`, `FR-3` — легко ссылаться
- **Чеклисты (checkboxes)** — человек видит прогресс, агент понимает статус
- **YAML frontmatter** — метаданные (статус, автор, приоритет)

Каждый тип артефакта использует общепринятый стандарт: OpenAPI для API, C4 + ADR для архитектуры, INVEST + BDD для stories, Gherkin для тестов. Полный список стандартов, шаблоны и примеры — в [docs/templates.md](docs/templates.md).

### Шаг 1. Планирование (PO)

**Вход:** Бизнес-требования, аналитика, фидбек
**Участники:** PO (ведёт)

**Артефакты:** Product Research Report → PRD → Product Backlog → User Stories

**Критерии готовности:** Каждая story имеет acceptance criteria, приоритеты проставлены, зависимости указаны.

### Шаг 2. Архитектура (Архитектор)

**Вход:** PRD, Product Backlog
**Участники:** Архитектор (ведёт), разработчики (консультируют)

**Артефакты:** Architecture Document, CONVENTIONS.md, Module Boundaries

CONVENTIONS.md — единый файл с правилами для всех агентов и разработчиков: стилевые правила, архитектурные ограничения, обязательные практики, запрещённые подходы. Обновляется один раз — применяется у всех.

**Критерии готовности:** Стек определён, модульные границы описаны, CONVENTIONS.md доступен.

### Шаг 3. Дизайн (UI/UX Дизайнер)

**Вход:** PRD, Product Backlog, бренд-гайды
**Участники:** Дизайнер (ведёт), PO (консультирует)

**Артефакты:** UI Specifications, Design Tokens, Interaction Specs

**Критерии готовности:** Все экраны описаны со всеми состояниями (empty, loading, error), компоненты привязаны к дизайн-системе, specs готовы для генерации кода.

### Шаг 4. Анализ (Системный аналитик)

**Вход:** PRD, Product Backlog, UI Specifications, Architecture Document
**Участники:** Аналитик (ведёт), разработчики (консультируют)

**Артефакты:** Task Specification, API Specification, Sequence Diagrams (Mermaid)

**Критерии готовности:** Edge cases покрыты, API контракты согласованы, спецификация прошла AI-валидацию.

### Шаг 5. Разработка (Разработчики)

**Вход:** Task Specs, UI Specs, API Specs
**Участники:** Разработчики (ведут), QA (параллельно пишет тесты)

**Артефакты:** Source Code, Unit Tests (≥80% покрытие), Pull Request

**Критерии готовности:** Код соответствует CONVENTIONS.md, тесты проходят, PR связан с задачей.

### Шаг 6. Code Review

**Вход:** Pull Request
**Участники:** Другой разработчик (reviewer)

**Процесс:**
1. **Агент** проверяет: стиль, lint, очевидные баги, покрытие тестами
2. **Человек** проверяет: архитектуру, бизнес-логику, edge cases
3. Только после обоих — merge

### Шаг 7. Тестирование (QA)

**Вход:** Готовые фичи, спецификации
**Участники:** QA (ведёт)

**Артефакты:** Test Report, Bug Reports, Coverage Analysis

### Шаг 8. Релиз

**Вход:** Протестированный код
**Участники:** Разработчики (деплой), QA (финальная проверка), PO (acceptance)

**Артефакты:** Release Notes (агент генерирует из коммитов), Changelog, Deployment Checklist

## 3. Контекст проекта

> **Формирование правильного контекста — самая важная задача.** Не только правильный промпт, но и правильный контекст. Плохой промпт с хорошим контекстом лучше, чем идеальный промпт без контекста.

### Двухуровневая документация

**Верхний уровень — общие знания о проекте:**

```
/
├── AGENTS.md              # Общие правила для всех агентов
├── ARCHITECTURE.md        # Архитектура проекта
├── CONVENTIONS.md         # Стилевые правила, паттерны
├── DESIGN-SYSTEM.md       # Дизайн-система
├── api-specs/             # API контракты
└── templates/             # Шаблоны артефактов
```

**Уровень модуля — детальные знания:**

```
src/features/
├── payments/
│   ├── AGENTS.md          # Контекст модуля payments
│   └── README.md
├── auth/
│   ├── AGENTS.md          # Контекст модуля auth
│   └── README.md
└── portfolio/
    ├── AGENTS.md
    └── README.md
```

### AGENTS.md (верхний уровень)

```markdown
# Project Agent Context

## Architecture
- Mobile: React Native
- Backend: Node.js + TypeScript
- Database: PostgreSQL
- API: REST + GraphQL

## Key Principles
- Feature-based architecture (каждый модуль самодостаточен)
- Strict typing (no any)
- Tests required for all new code

## Module Boundaries
- payments/ — ФИО разработчика (контакт)
- auth/ — ФИО разработчика (контакт)
- portfolio/ — ФИО разработчика (контакт)

## DON'T
- Не трогай модули вне своей зоны без согласования
- Не создавай новые shared-утилиты без обсуждения
- Не меняй design tokens без согласования с дизайнером
```

### AGENTS.md (уровень модуля)

```markdown
# Payments Module

## Purpose
Модуль оплаты: счета, переводы, история транзакций.

## Architecture
- services/ — бизнес-логика
- routes/ — API endpoints
- models/ — типы данных
- tests/ — тесты

## Key Files
- services/payment.service.ts — основная логика
- routes/payment.routes.ts — API
- models/transaction.ts — модель транзакции

## Dependencies
- auth/ — для авторизации (только через auth.service)
- portfolio/ — для получения портфеля (только через portfolio.service)

## Current Tasks
- [ ] Добавить поддержку рекуррентных платежей
- [ ] Рефакторинг валидации
```

### Как ставить задачу в промпте

1. **Ссылка на спецификацию** — «Реализуй TS-42» (агент читает шаблон)
2. **Границы** — «Работай только в src/features/payments/»
3. **Критерии готовности** — «Должны проходить тесты, покрытие ≥80%»
4. **Что НЕ делать** — «Не трогай auth middleware, не создавай новых файлов в shared/»

### Управление конфликтами

Когда несколько разработчиков дают агентам задачи в одной кодовой базе:

1. **Границы по модулям** — каждый разработчик закреплён за модулем, указано в AGENTS.md
2. **Единая дизайн-система** — все работают с одними и теми же tokens и компонентами
3. **Единый CONVENTIONS.md** — один файл с правилами для всех агентов

### Конфигурация агентов из репозитория

Все агенты используют единую конфигурацию из репо:

```
.codex/
├── instructions.md        # Общие инструкции
├── conventions.md         # Стилевые правила
└── architecture.md        # Архитектурные ограничения
```

> **`.codex/` vs `.agents/skills/`:**
> - `.codex/` — **конфигурация** агента (инструкции, правила). Единая для всех, задаётся архитектором. «Как работать».
> - `.agents/skills/` — **навыки** по ролям. Персональные, настраиваются Role Lead. «Что уметь делать».

### Шаблон проекта

При старте нового проекта:

```
project-repo/
├── AGENTS.md                  # Контекст для всех агентов
├── ARCHITECTURE.md            # Архитектура проекта

├── DESIGN-SYSTEM.md           # Дизайн-система

├── .codex/                    # Конфигурация агентов
│   ├── instructions.md
│   ├── conventions.md
│   └── architecture.md
│

│   ├── po/
│   ├── architect/
│   ├── designer/
│   ├── analyst/
│   ├── developer/
│   └── qa/
│
├── templates/                 # Шаблоны артефактов (см. docs/templates.md)
│
├── docs/                      # Документация
│   ├── onboarding/
│   │   ├── codex-guide.md     # Гайд по инструменту для команды
│   │   ├── prompt-guide.md    # Как ставить задачу агенту
```

## 4. Agent Skills

[agentskills.io](https://agentskills.io) — единый открытый формат для навыков AI-агентов. Поддерживается Codex, Claude Code, Gemini CLI, Cursor.

**Формат:** директория с обязательным `SKILL.md`:

```
skill-name/
├── SKILL.md          # Метаданные + инструкции
├── scripts/          # Исполняемые скрипты
├── references/       # Документация
└── assets/           # Шаблоны, ресурсы
```

### Принцип: готовые → адаптировать → создавать свои

1. **Подтянуть готовые** skills из каталогов (PO, дизайнер, разработчик — хорошо покрыты)
2. **Адаптировать** под специфику проекта (свои шаблоны, контекст)
3. **Создать свои** где готовых нет (архитектор, аналитик, часть QA)

Полный каталог skills по ролям, источники и команды установки — в [docs/skills-catalog.md](docs/skills-catalog.md).

## 5. Внедрение: поэтапный план

### Принцип: постепенное ускорение, а не революция

**Фаза 1: Знакомство (недели 1-2)**
- Каждый начинает использовать AI-агента как «autocomplete на стероидах»
- Без изменений в процессе — просто индивидуальная продуктивность
- Цель: снять страх, набить руку

**Фаза 2: Структура (недели 3-4)**
- Внедрение общих `.codex/` инструкций
- Договорённость о правилах (CONVENTIONS.md)
- Назначаются Role Leads по каждой специализации

**Фаза 3: Процесс (месяц 2)**
- Переход на AI-friendly артефакты
- Agent-first подход: сначала агент генерирует, потом человек ревьюит
- Code review с агентным pre-pass

**Фаза 4: Оптимизация (месяц 3+)**
- Команда сама видит, где агент помогает, где мешает
- Доработка шаблонов и инструкций на основе опыта
- Расширение зон ответственности агентов

### Онбординг команды

**Двойной онбординг:**

1. **В инструмент** — концепция, как ставить задачу агенту, как review-ить результат
2. **В процесс** — артефакты, форматы, границы ответственности модуля

| Неделя | Что делаем | Результат |
|---|---|---|
| 1 | Знакомство с инструментом: концепция, интерфейс | Каждый умеет ставить задачу агенту |
| 2 | Практика на реальных задачах (под наблюдением) | Первый опыт, первые ошибки |
| 3 | Изучение артефактов и шаблонов проекта | Понимание форматов и связей |
| 4 | Самостоятельная работа с агентом | Полноценная работа в процессе |

**Role Lead** — назначается на каждую роль, отвечает за стандарты, онбординг и качество артефактов в своей зоне.

### Анти-паттерны

Типичные ошибки при внедрении:

**Процесс:**

| Анти-паттерн | Последствия | Как избежать |
|---|---|---|
| AI-first вместо AI-assisted | Некачественный код, пропущенные требования | Агент генерирует черновик, человек ревьюит |
| Пропуск стандартизации | 10 агентов = 10 стилей | Сначала AGENTS.md + CONVENTIONS.md, потом запуск |
| Контекст не сформирован | Агент в отрыве от реальности | AGENTS.md на каждом уровне, ссылки в промптах |
| Слишком много автоматизации сразу | Команда теряет контроль | Поэтапное внедрение |

**Люди:**

| Анти-паттерн | Как избежать |
|---|---|
| Слепое доверие агенту | Обязательный human review |
| Страх перед агентом | Онбординг, менторство |
| Делегирование ответственности агенту | Человек всегда отвечает за финальный артефакт |
| Секретность промптов | Промпты — часть team knowledge, хранятся в `.agents/skills/` |

**Технический:**

| Анти-паттерн | Как избежать |
|---|---|
| Агент за пределами зоны | Границы в AGENTS.md, правила в промптах |
| Игнорирование security | Агент — только dev, прод через CI/CD |
| Агент без ограничений | Таймауты, лимиты, подтверждение действий |

## 6. Workflow по ролям

Краткий обзор — каждый workflow показывает, как роль работает с агентом на протяжении SDLC. Детальные примеры с промптами — в [docs/workflows.md](docs/workflows.md).

### PO: Product Research → PRD → Backlog → Приоритеты

```
Product Research → PRD → Backlog → Приоритизированный Backlog
      ↓               ↓         ↓            ↓
   (PO+agent)    (PO+agent) (PO+agent)  (PO+agent)
```

PO использует агента на каждом этапе: исследование рынка, формулирование vision, декомпозиция, приоритизация. Всегда ревьюит результат перед передачей дальше.

### Архитектор: Architecture → Standards → Boundaries → Review

```
Architecture Doc → CONVENTIONS.md → Module Boundaries → Continuous Review
        ↓               ↓                  ↓                    ↓
  (Arch+agent)    (Arch+agent)       (Arch+agent)        (Arch+agent)
```

Архитектор проектирует один раз, но reviewит непрерывно. CONVENTIONS.md — главный артефакт, синхронизирующий всех разработчиков.

### Дизайнер: Design System → UI Specs → Code

```
Design System → UI Specifications → Конвертация в код
      ↓                ↓                    ↓
 (Des+agent)     (Des+agent)          (Des+agent)
```

### Аналитик: Stories → Task Specs → API Contracts → Validation

```
User Stories → Task Specifications → API Contracts → Валидация
      ↓              ↓                     ↓               ↓
 (Anal+agent)   (Anal+agent)          (Anal+agent)    (Anal+agent)
```

## 7. Метрики успеха

### Принцип измерения

1. Замерить **baseline ДО** внедрения (1-2 спринта без AI)
2. Замерять **после каждой фазы** внедрения
3. Сравнивать **тренды**, а не абсолютные числа

### Скорость доставки

| Метрика | Что измеряет | Как замерять |
|---|---|---|
| Lead Time | Время от идеи до продакшена | created → deployed |
| Cycle Time | Время от начала до готовности фичи | in progress → done |
| Throughput | Завершённые stories за спринт | count per sprint |
| Deployment Frequency | Частота релизов | CI/CD статистика |

**Ожидание с AI:** рост throughput на 30-50%.

### Качество

| Метрика | Что измеряет | Как замерять |
|---|---|---|
| Bug Rate | Багов на фичу | прод-баги / фичи |

| Test Coverage | Покрытие автотестами | CI/CD: coverage |
| PR Rejection Rate | Доля возвращённых PR | отклонённые / все |

**Ожидание с AI:** снижение bug rate за счёт агентного pre-review.

## 8. Масштабирование

### Что меняется при росте

| Аспект | 10 человек | 20+ человек | 50+ человек |
|---|---|---|---|
| Модули | 5-6, каждый за своим | 10-15, submodule owners | 20+, team-based |
| Контекст | Все знают проект | Никто не знает весь | PROJECT-MAP.md обязателен |
| Роли | Один на роль | Дублирование | Role Leads + гильдии |
| Code Review | Peer-to-peer | Агентный pre-review обязателен | Многоуровневый |
| Координация | Standup + чат | Sprint ceremonies + Role Leads | Tribe/Squad |

### Module Owners

При росте каждый модуль получает owner'а: отвечает за AGENTS.md модуля, контролирует интерфейсы, проводит cross-module review.

### Role Leads

При 2+ человек в роли назначается Role Lead: отвечает за standards, онбординг, консистентность артефактов.

### PROJECT-MAP.md

При 20+ человек — навигационный документ с картой модулей, owners и связями.

### Многоуровневый Code Review

| Уровень | Кто | Что проверяет |
|---|---|---|
| 1. Агентный pre-review | AI | Стиль, lint, баги, покрытие |
| 2. Peer review | Другой разработчик | Бизнес-логика, edge cases |
| 3. Module owner review | Owner модуля | Cross-module влияние |
| 4. Архитектурный review | Архитектор | Shared code, API контракты |

### Новые роли при масштабировании

| Роль | Когда | Зачем агент |
|---|---|---|
| Tech Lead | 20+ | Архитектурные review, мониторинг техдолга |
| Engineering Manager | 20+ | Метрики команды, планирование capacity |
| DevOps / SRE | 20+ | CI/CD, мониторинг, инциденты |
