# AI-Augmented Team Transformation Guide

Руководство по внедрению AI-агентов в работу команды разработки цифрового продукта.

## Контекст

Команда из 10 человек (разработчики, дизайнеры, системные аналитики, архитектор, продакт-оунер, QA) работает над цифровым продуктом. Команда привыкла к традиционному SDLC без AI-инструментов. У команды нет опыта использования AI-агентов.

## Цель

Кратно ускорить рабочий процесс команды с помощью AI-агентов, не разрушив то, что уже работает. Дать каждому члену команды персонального AI-агента, который усилит его экспертизу и повысит продуктивность.

## Ключевой принцип

**Агент не заменяет экспертизу — агент расширяет и усиливает её.**

Каждый член команды становится «tech lead» для своего AI-агента: направляет, проверяет, принимает решения. Агент берёт рутину и ускоряет execution.

## 1. Роли и зоны ответственности

Каждая роль имеет **двойную ответственность:**

1. **Развивать свой инструментарий** — настраивать, обучать, адаптировать один или несколько AI-агентов (навыков) под задачи своей специализации, повышая эффективность своей роли в команде
2. **Производить качественный AI-friendly артефакт** — результат работы роли должен быть в формате, удобном для агентов других ролей, и пополнять общий контекст проекта

### Продакт-оунер (PO)

| Что делает | Как использует агента | Артефакт на выход |
|---|---|---|
| Продуктовое исследование | Агент помогает с конкурентным анализом, формированием гипотез, исследованием рынка | Product Research Report |
| Формулирует vision проекта | Агент помогает структурировать идеи, дополнить требования | PRD (Product Requirements Document) |
| Декомпозирует PRD на фичи и user stories | Агент помогает разбить на user stories, найти пропуски | Product Backlog |
| Приоритизирует бэклог | Агент помогает оценить impact, найти зависимости | Приоритизированный Backlog |

### Архитектор

| Что делает | Как использует агента | Артефакт на выход |
|---|---|---|
| Проектирует архитектуру проекта | Агент помогает оценить технологический стек, предлагает архитектурные паттерны | Architecture Document |
| Формирует архитектурные стандарты | Агент помогает выявить паттерны, анти-паттерны, Best Practices | Architectural Guidelines (CONVENTIONS.md) |
| Определяет модульные границы | Агент помогает выделить модули, определить интерфейсы между ними | Module Boundaries |
| Контролирует архитектурную целостность | Агент проверяет PR на соответствие архитектурным стандартам | Architecture Review |

### Системный аналитик

| Что делает | Как использует агента | Артефакт на выход |
|---|---|---|
| Декомпозирует user stories в технические спецификации | Агент проверяет полноту, предлагает edge cases, валидирует against архитектуры | Task Specification (structured) |
| Описывает API, контракты, бизнес-логику | Агент генерирует схемы, примеры, проверяет консистентность | API Specification |

### UI/UX Дизайнер

| Что делает | Как использует агента | Артефакт на выход |
|---|---|---|
| Проектирует интерфейсы | Агент конвертирует макеты в код, поддерживает консистентность дизайн-системы | Design System + UI Specs |
| Описывает interaction patterns | Агент помогает с accessibility, предлагает варианты для edge cases | Component Library Spec |

### Разработчик

| Что делает | Как использует агента | Артефакт на выход |
|---|---|---|
| Реализует задачи по спецификации | Агент пишет boilerplate, тесты, рефакторит, ищет баги | Source Code + Unit Tests |
| Code review | Агент делает первый проход (lint, стиль, очевидные баги) | Review Comments |

### QA-инженер

| Что делает | Как использует агента | Артефакт на выход |
|---|---|---|
| Проектирует тест-стратегию | Агент генерирует тест-кейсы по спецификации | Test Plan + Test Cases |
| Пишет автотесты | Агент генерирует автотесты, анализирует coverage | Automated Tests |

## 2. SDLC: шаги и артефакты

Каждый шаг produces чёткий артефакт в AI-friendly формате. Артефакт становится частью контекста проекта.

### Шаг 1. Планирование (PO)

**Вход:** Бизнес-требования, аналитика, фидбек
**Участники:** PO (ведёт)
**Агенты:** PO использует своих агентов

**Артефакты:**
- Product Research Report — результаты исследования рынка
- PRD — Product Requirements Document (vision проекта)
- Product Backlog — структурированный, с приоритетами, acceptance criteria
- User Stories — пронумерованные, связанные с эпиками

**Критерии готовности:**
- Каждая story имеет чёткие acceptance criteria
- Приоритеты проставлены
- Зависимости между stories указаны

### Шаг 2. Архитектура (Архитектор)

**Вход:** PRD, Product Backlog
**Участники:** Архитектор (ведёт), разработчики (консультируют)
**Агент:** Помогает оценить технологический стек, предлагает архитектурные паттерны, выделяет модули

**Артефакты:**
- Architecture Document — общая архитектура проекта, технологический стек, обоснование выбора
- Architectural Guidelines (CONVENTIONS.md) — стилевые правила, паттерны, анти-паттерны, обязательные практики
- Module Boundaries — структура модулей, интерфейсы между ними, зоны ответственности

**Критерии готовности:**
- Технологический стек определён и обоснован
- Модульные границы определены, интерфейсы описаны
- Архитектурные стандарты согласованы с командой
- CONVENTIONS.md доступен всем агентам разработчиков

### Шаг 3. Дизайн (UI/UX Дизайнер)

**Вход:** PRD, Product Backlog, бренд-гайды
**Участники:** Дизайнер (ведёт), PO (консультирует)
**Агент:** Конвертирует в код, проверяет консистентность с дизайн-системой

**Артефакты:**
- UI Specifications — структура экранов, компоненты, состояния
- Design Tokens — цвета, типографика, spacing (в коде)
- Interaction Specs — переходы, анимации, edge states

**Критерии готовности:**
- Все экраны описаны с учётом всех состояний (empty, loading, error)
- Компоненты привязаны к дизайн-системе
- Specs в формате, пригодном для генерации кода

### Шаг 4. Анализ (Системный аналитик)

**Вход:** PRD, Product Backlog, UI Specifications, Architecture Document
**Участники:** Аналитик (ведёт), разработчики (консультируют)
**Агент:** Помогает найти пропуски, edge cases, валидирует against архитектуры

**Артефакты:**
- Task Specification — детальная техническая спецификация по каждой story
- API Specification — контракты, схемы данных
- Sequence Diagrams — в формате Mermaid

**Критерии готовности:**
- Все edge cases покрыты
- API контракты согласованы с разработчиками
- Спецификация прошла AI-валидацию на полноту

### Шаг 5. Разработка (Разработчики)

**Вход:** Task Specs, UI Specs, API Specs
**Участники:** Разработчики (ведут), QA (параллельно пишет тесты)
**Агент:** Пишет код по спецификации, boilerplate, тесты

**Артефакты:**
- Source Code — в рамках границ ответственности
- Unit Tests — покрытие ≥80% нового кода
- Pull Request — с описанием, ссылкой на спецификацию

**Критерии готовности:**
- Код соответствует стилевым правилам проекта
- Тесты проходят
- PR связан с задачей

### Шаг 6. Code Review

**Вход:** Pull Request
**Участники:** Другой разработчик (reviewer)
**Агент:** Предварительный review (автоматический)

**Артефакты:**
- Agent Review — автоматический первый проход
- Human Review — фокус на архитектуре и бизнес-логике

**Процесс:**
1. Агент проверяет: стиль, lint, очевидные баги, покрытие тестами
2. Человек проверяет: архитектуру, бизнес-логику, edge cases
3. Только после обоих — merge

### Шаг 7. Тестирование (QA)

**Вход:** Готовый фичи, спецификации
**Участники:** QA (ведёт)
**Агент:** Генерирует тест-кейсы, автотесты, анализирует coverage

**Артефакты:**
- Test Report — что проверено, что нет
- Bug Reports — воспроизводимые, с шагами
- Coverage Analysis — что покрыто автотестами

### Шаг 8. Релиз

**Вход:** Протестированный код
**Артефакты:**
- Release Notes (агент генерирует из коммитов)
- Changelog
- Deployment Checklist

## 3. Шаблоны артефактов

Все шаблоны фиксируются один раз на старте проекта и хранятся в репозитории проекта в директории `templates/`.

### Принципы AI-friendly + human-readable формата

Каждый артефакт должен быть удобен **и для людей, и для AI-агентов**. Вот как этого достичь:

1. **Структурированный markdown** — человек читает как документ, агент парсит по секциям
2. **Обязательные заголовки** — агент знает где искать нужные данные
3. **Связи через идентификаторы** — `US-42`, `TS-15`, `FR-3` — и человек и агент могут ссылаться на конкретные элементы
4. **Чеклисты (checkboxes)** — человек видит прогресс, агент понимает статус готовности
5. **Минимум свободного текста в критичных полях** — ключевая информация в структурированных полях, контекст и пояснения — в свободном тексте
6. **YAML frontmatter** — метаданные (статус, автор, приоритет, связи) — агент парсит мгновенно, человек видит шапку документа

### Стандарты для типов артефактов

Каждый тип артефакта должен использовать общепринятый стандарт. Это обеспечивает консистентность, совместимость инструментов и понятность для всех участников (людей и агентов).

| Тип артефакта | Рекомендуемый стандарт | Описание |
|---|---|---|
| API Specification | OpenAPI 3.x (Swagger) | Стандарт описания REST API. Агент может генерировать код, тесты и документацию из спецификации |
| Technical Specification | OpenSpec (Spec-Driven Development) | [openspec.dev](https://openspec.dev) — фреймворк для спецификаций, оптимизированный для AI-агентов. Spec delta показывает изменения требований |
| Architecture Document | C4 Model + ADR | C4 для визуализации архитектуры (Mermaid), ADR (Architecture Decision Records) для решений |
| User Stories | INVEST + BDD (Given/When/Then) | INVEST критерии качества stories, BDD формат для acceptance criteria |
| UI Specification | Design Tokens (W3C) + Figma API | Стандарт W3C для design tokens, Figma API для программного доступа к дизайн-системе |
| Test Cases | Gherkin (BDD) | Общепринятый формат для тест-кейсов: Feature / Scenario / Given-When-Then |
| Code Conventions | EditorConfig + ESLint/Prettier configs | Автоматически проверяемые правила форматирования и стиля |
| PR Description | Conventional Commits | Стандарт формата коммитов и PR (feat:, fix:, docs: и т.д.) |

### Принцип: стандарт → шаблон → автоматизация

1. Выбрать стандарт для типа артефакта
2. Создать шаблон на основе стандарта (в `templates/`)
3. Настроить агента на использование шаблона
4. По возможности — автоматизировать валидацию (lint, schema validation)

### Требования к шаблонам:

- **Структурированные** — agent может парсить и заполнять
- **Пронумерованные секции** — легко ссылаться
- **Обязательные поля** — agent знает, что заполнить
- **Связи между артефактами** — каждая спецификация ссылается на story, каждый PR — на спецификацию

### Минимальный набор шаблонов:

```

├── product-research.md    # Отчёт о продуктовом исследовании
├── prd.md                 # Product Requirements Document
```

## 4. Контекст для агентов

**Формирование правильного контекста — самая важная задача при работе с AI-агентами.** Не только правильный промпт, но и правильный контекст.

### Принцип: двухуровневая документация

**Верхний уровень — общие знания о проекте:**
```
/
├── AGENTS.md              # Общие правила для всех агентов
├── ARCHITECTURE.md        # Архитектура проекта
├── CONVENTIONS.md         # Стилевые правила, паттерны
├── DESIGN-SYSTEM.md       # Дизайн-система
├── api-specs/             # API контракты
└── templates/             # Шаблоны артефактов
```

**Уровень модуля — детальные знания:**
```
src/
├── features/
│   ├── payments/
│   │   ├── AGENTS.md      # Контекст модуля payments
│   │   ├── README.md      # Описание модуля
│   │   └── ...
│   ├── auth/
│   │   ├── AGENTS.md      # Контекст модуля auth
│   │   ├── README.md
│   │   └── ...
│   └── portfolio/
│       ├── AGENTS.md
│       ├── README.md
│       └── ...
```

### Что должно быть в AGENTS.md (верхний уровень):

```markdown
# Project Agent Context

## Architecture
- Mobile: React Native
- Backend: Node.js + TypeScript
- Database: PostgreSQL
- API: REST + GraphQL

## Key Principles
- Feature-based architecture (каждый модуль самодостаточен)
- Strict typing (no any)
- Tests required for all new code

## Module Boundaries
- payments/ — ФИО разработчика (контакт)
- auth/ — ФИО разработчика (контакт)
- portfolio/ — ФИО разработчика (контакт)

## DON'T
- Не трогай модули вне своей зоны без согласования
- Не создавай новые shared-утилиты без обсуждения
- Не меняй design tokens без согласования с дизайнером
```

### Что должно быть в AGENTS.md (уровень модуля):

```markdown
# Payments Module

## Purpose
Модуль оплаты: счета, переводы, история транзакций.

## Architecture
- services/ — бизнес-логика
- routes/ — API endpoints
- models/ — типы данных
- tests/ — тесты

## Key Files
- services/payment.service.ts — основная логика
- routes/payment.routes.ts — API
- models/transaction.ts — модель транзакции

## Dependencies
- auth/ — для авторизации (только через auth.service)
- portfolio/ — для получения портфеля (только через portfolio.service)

## Current Tasks
- [ ] Добавить поддержку рекуррентных платежей
- [ ] Рефакторинг валидации
```

### Как ставить задачу в промпте:

**Фокус:** контекст > промпт. Плохой промпт с хорошим контекстом лучше, чем идеальный промпт без контекста.

**Структура задачи для агента:**
1. **Ссылка на спецификацию** — «Реализуй TS-42» (агент читает шаблон)
2. **Границы** — «Работай только в src/features/payments/»
3. **Критерии готовности** — «Должны проходить тесты, покрытие ≥80%»
4. **Что НЕ делать** — «Не трогай auth middleware, не создавай новых файлов в shared/»

## 5. Управление конфликтами

### Проблема: Конфликты агентов

Когда 10 разработчиков дают своим агентам задачи в одной кодовой базе:

**Решение 1: Границы ответственности по модулям**

Каждый разработчик закреплён за модулем:
- Разработчик A → `features/payments/`
- Разработчик B → `features/auth/`
- Разработчик C → `features/portfolio/`

В AGENTS.md каждого модуля указано, кто ответственный.

**Решение 2: Общий контекст через дизайн-систему**

Несмотря на разделение по модулям, все разработчики работают с единой дизайн-системой и API-контрактами. Это обеспечивает консистентность.

**Решение 3: Единые правила разработки (CONVENTIONS.md)**

Архитектор формирует `CONVENTIONS.md` на шаге «Архитектура»:
- Стилевые правила (naming, formatting)
- Архитектурные ограничения (паттерны, анти-паттерны)
- Обязательные практики (typing, testing, error handling)
- Запрещённые подходы

Этот артефакт синхронизируется со всеми разработчиками и встроен в конфиг каждого агента.

### Проблема: 10 агентов = 10 стилей кода

**Решение:** Конфигурация агентов берётся из репозитория:

```
.codex/
├── instructions.md        # Общие инструкции для Codex
├── conventions.md         # Стилевые правила
└── architecture.md        # Архитектурные ограничения
```

Все разработчики используют одну и ту же конфигурацию из репо. Обновил один раз — изменилось у всех.

## 6. Онбординг команды

### Двойной онбординг:

**1. Онбординг в AI-инструмент (Codex):**
- Концепция инструмента: какие примитивы есть (роли, skills, агенты)
- Как ставить задачу агенту: структура промпта, фокус на контексте
- Как review-ить результат агента: что проверять, где агент ошибается
- Практика: простые задачи под наблюдением

**2. Онбординг в процесс (SDLC с AI):**
- Какие артефакты, в каком формате, когда
- Как читать артефакты от других ролей
- Как писать AI-friendly артефакты
- Границы ответственности своего модуля

### План онбординга:

| Неделя | Что делаем | Результат |
|---|---|---|

| 2 | Практика на реальных задачах (под наблюдением) | Первый опыт, первые ошибки |
| 3 | Изучение артефактов и шаблонов проекта | Понимание форматов и связей |
| 4 | Самостоятельная работа с агентом | Полноценная работа в процессе |

### Менторство:

На каждую роль назначается **Role Lead** (на старте проекта), который:
- Формирует общие правила и ограничения для своей роли
- Помогает онбордить новых членов команды
- Обновляет шаблоны и инструкции по мере развития проекта

## 8. Внедрение: поэтапный план

### Принцип: постепенное ускорение, а не революция

**Фаза 1: Знакомство (недели 1-2)**
- Каждый начинает использовать Codex как «autocomplete на стероидах»
- Без изменений в процессе — просто индивидуальная продуктивность
- Цель: снять страх, набить руку

**Фаза 2: Структура (недели 3-4)**
- Внедрение общих `.codex/` инструкций
- Договорённость о правилах (CONVENTIONS.md)
- Назначаются Role Leads по каждой специализации

**Фаза 3: Процесс (месяц 2)**
- Переход на AI-friendly артефакты
- Agent-first подход: сначала агент генерирует, потом человек ревьюит
- Code review с агентным pre-pass

**Фаза 4: Оптимизация (месяц 3+)**
- Команда сама видит, где агент помогает, где мешает
- Доработка шаблонов и инструкций на основе опыта
- Расширение зон ответственности агентов

### Чего НЕ делать:

- ❌ Не давать агенту доступ к проду напрямую
- ❌ Не пускать агентов в чужие модули без координации
- ❌ Не заменять human review на агентный (только дополнение)
- ❌ Не начинать с полной автоматизации
- ❌ Не игнорировать контекст — «правильный промпт» не работает без правильного контекста

## 10. Пример: рабочий день PO с AI-агентом

Показываем пошагово, как продакт-оунер работает с AI-агентом на протяжении полного цикла SDLC. Этот пример можно адаптировать для других ролей.

---

### Шаг 1. Продуктовое исследование

**Задача:** Изучить рынок, конкурентов, сформулировать гипотезы

**Skill:** `competitive-analysis`

**Пример промпта:**
```
Исследуй рынок мобильных банковских приложений в России.
Основные конкуренты: Тинькофф, Альфа, ВТБ.
Нас интересует: UX онбординга, фичи портфеля, подписочные модели.
Результат — Product Research Report по шаблону templates/product-research.md
```

**Разделение труда:**
- Агент: собирает данные, структурирует, выделяет паттерны
- PO: оценивает релевантность, добавляет экспертное мнение, формулирует гипотезы

**Артефакт:** Product Research Report
**Передаётся:** используется PO для формирования PRD

---

### Шаг 2. Формирование vision (PRD)

**Задача:** Описать видение продукта, бизнес-требования, целевую аудиторию

**Skill:** `jobs-to-be-done`

**Пример промпта:**
```
На основе Product Research Report (docs/product-research.md)
помоги сформулировать PRD для инвестиционного мобильного приложения.
Целевая аудитория: начинающие инвесторы 25-40 лет.
Ключевое отличие: упрощённый UX + AI-рекомендации.
Результат — PRD по шаблону templates/prd.md
```

**Разделение труда:**
- Агент: структурирует идеи, дополняет разделы (NFR, риски, KPI), проверяет полноту
- PO: формулирует vision, определяет MVP scope, принимает решения о trade-offs

**Артефакт:** PRD (Product Requirements Document)
**Передаётся:** Архитектору (шаг SDLC 2), Дизайнеру (шаг SDLC 3)

---

### Шаг 3. Декомпозиция в User Stories

**Задача:** Разбить PRD на конкретные user stories с acceptance criteria

**Skills:** `brainstorm-stories` + `write-user-story`

**Пример промпта:**
```
На основе PRD (docs/prd.md) декомпозируй продукт на user stories.
Используй формат INVEST, acceptance criteria в BDD (Given/When/Then).
Группируй по эпикам: Онбординг, Портфель, Торговля, Аналитика.
Результат — backlog по шаблону templates/user-story.md
```

**Разделение труда:**
- Агент: генерирует черновики stories, предлагает acceptance criteria, находит пропуски
- PO: проверяет каждую story, корректирует приоритеты, отсекает лишнее

**Артефакт:** Product Backlog (набор user stories)
**Передаётся:** PO для приоритизации

---

### Шаг 4. Приоритизация бэклога

**Задача:** Расставить приоритеты, найти зависимости

**Skill:** `agile-product-owner`

**Пример промпта:**
```
Помоги приоритизировать backlog (docs/backlog.md).
Учитывай: бизнес-ценность (high/medium/low), техническую сложность,
зависимости между stories. Предложи порядок реализации для MVP.
```

**Разделение труда:**
- Агент: анализирует зависимости, предлагает приоритеты, оценивает риски
- PO: принимает финальное решение, учитывая бизнес-контекст не доступный агенту

**Артефакт:** Приоритизированный Backlog
**Передаётся:** Аналитику (после Архитектуры и Дизайна)

---

### Итог workflow PO

```
Product Research → PRD → Backlog → Приоритизированный Backlog
      ↓               ↓         ↓            ↓
   (PO+agent)    (PO+agent) (PO+agent)  (PO+agent)
                   ↓    ↓                  ↓
              Архитектор  Дизайнер     Аналитик
```

**Ключевые принципы workflow:**
- PO всегда ревьюит результат агента перед передачей дальше
- Каждый артефакт создаётся по шаблону из `templates/`
- Артефакты связаны: PRD → Stories → Приоритеты
- Промпт содержит ссылку на предыдущий артефакт (контекст)

## 11. Метрики успеха

Как понять, что внедрение AI-агентов дало результат, а не просто добавило overhead.

### Принцип измерения

1. Замерить **baseline ДО** внедрения (1-2 спринта без AI)
2. Замерять **после каждой фазы** внедрения
3. Сравнивать **тренды**, а не абсолютные числа

### Скорость доставки (Delivery Metrics)

| Метрика | Что измеряет | Как замерять |
|---|---|---|
| Lead Time | Время от идеи до продакшена | Jira/трекер: created → deployed |
| Cycle Time | Время от начала работы до готовности фичи | Jira: in progress → done |
| Throughput | Количество завершённых stories за спринт | Story points или count per sprint |
| Deployment Frequency | Как часто релизим | CI/CD статистика |

**Ожидание с AI:** рост throughput на 30-50% без роста команды.

### Качество (Quality Metrics)

| Метрика | Что измеряет | Как замерять |
|---|---|---|
| Bug Rate | Количество багов на фичу | Баги в проде / фичи за спринт |
| Code Review Time | Время на review | PR: open → approved |
| Test Coverage | Покрытие автотестами | CI/CD: coverage report |
| PR Rejection Rate | Доля PR, возвращённых после review | Отклонённые PR / все PR |

**Ожидание с AI:** снижение bug rate за счёт агентного pre-review и автотестов.

### Продуктивность по ролям

| Роль | Метрика | Что измеряет |
|---|---|---|
| PO | Time to PRD | Время от идеи до готового PRD |
| PO | Time to Backlog | Время от PRD до приоритизированного backlog |
| Архитектор | Time to Architecture | Время на формирование Architecture Document |
| Дизайнер | Time to UI Spec | Время от PRD до готовых UI Specs |
| Аналитик | Time to Task Spec | Время от backlog + дизайна до Task Specification |
| Разработчик | Time to PR | Время от спецификации до готового Pull Request |
| QA | Time to Test Plan | Время от спецификации до готового test plan |

**Ожидание с AI:** сокращение времени на каждом этапе в 2-3x за счёт генерации черновиков агентом.

### Вовлечённость команды (Adoption)

| Метрика | Что измеряет | Как замерять |
|---|---|---|
| Agent Usage Rate | Процент команды, регулярно использующих агента | Опрос / логи инструментов |
| Team Satisfaction | Удовлетворённость работой с AI | Короткий опрос раз в спринт (1-5) |
| Custom Skills Created | Количество созданных кастомных skills | Количество skills в `.agents/skills/` |
| Self-Improvement | Самостоятельное улучшение агентных навыков | Количество обновлений templates/skills за спринт |

**Цель:** Agent Usage Rate ≥80% к концу месяца 2, Satisfaction ≥4/5.

## 12. Анти-паттерны

Типичные ошибки команд при внедрении AI-агентов.

### Процесс

| Анти-паттерн | Последствия | Как избежать |
|---|---|---|
| AI-first вместо AI-assisted | Некачественный код, пропущенные требования, баги в проде | Агент генерирует черновик, человек ревьюит и принимает решения |
| Пропуск стандартизации | 10 агентов = 10 стилей, артефакты не связаны, хаос | Сначала AGENTS.md + CONVENTIONS.md + templates/, потом запуск |
| Контекст не сформирован | Агент генерирует код/документы в отрыве от реальности | AGENTS.md на каждом уровне, ссылки на артефакты в промптах |
| Слишком много автоматизации сразу | Команда теряет контроль, не понимает что делает агент | Поэтапное внедрение (см. раздел 8) |

### Люди

| Анти-паттерн | Последствия | Как избежать |
|---|---|---|
| Слепое доверие агенту | Баги, архитектурные нарушения | Обязательный human review, агент — только первый проход |
| Страх перед агентом | Разделение на «использующих» и «нет», неравный вклад | Онбординг, менторство, демонстрация реальной пользы |
| Делегирование ответственности агенту | Человек снимает с себя ответственность за результат | Правило: человек всегда отвечает за финальный артефакт |
| Секретность промптов | Никто не учится на чужих ошибках | Промпты — часть team knowledge, хранятся в `.agents/skills/` |

### Технический

| Анти-паттерн | Последствия | Как избежать |
|---|---|---|
| Агент за пределами зоны ответственности | Конфликты, сломанный код чужого модуля | Границы в AGENTS.md, правила в промптах |
| Игнорирование security | Утечки, поломка продакшена | Агент — только dev-окружение, доступ к проду через CI/CD |
| Агент без ограничений | Непредсказуемые изменения, длительные выполнения | Таймауты, ограничения на файлы, обязательное подтверждение действий |

## 13. Масштабирование

Что меняется при росте команды.

### Что меняется при росте

| Аспект | 10 человек | 20+ человек | 50+ человек |
|---|---|---|---|
| Модули | 5-6, каждый за своим | 10-15, появляются submodule owners | 20+, team-based ownership |
| Контекст | Все знают весь проект | Никто не знает весь проект | PROJECT-MAP.md обязателен |
| Роли | Один человек на роль | Дублирование ролей | Role Leads + гильдии |
| Code Review | Peer-to-peer | Агентный pre-review обязателен | Многоуровневый review |
| Онбординг | На коленке | Формализованный | Структурированная программа |
| Координация | Standup + чат | Sprint ceremonies + Role Leads | Tribe/Squad структура |

### Ключевые изменения при масштабировании

**Module Owners**

При росте команды каждый модуль получает owner'а, который:
- Отвечает за AGENTS.md своего модуля
- Контролирует изменения интерфейсов модуля
- Проводит архитектурный review для cross-module PR

**Role Leads**

При появлении 2+ человек в одной роли назначается Role Lead:
- Отвечает за standards и templates для своей роли
- Проводит онбординг новых членов в роль
- Обеспечивает консистентность артефактов внутри роли
- Организует peer review внутри специализации

**PROJECT-MAP.md**

При 20+ человек появляется навигационный документ:
```markdown
# Project Map

## Модули
| Модуль | Owner | Команда | Описание |
|---|---|---|---|
| payments/ | Иванов А. | Backend-1 | Платежи и транзакции |
| auth/ | Петров Б. | Backend-2 | Авторизация |
| portfolio/ | Сидоров В. | Backend-1 | Портфель и аналитика |

## API контракты
- payments ↔ auth: auth.service.validate()
- portfolio ↔ payments: payment.service.getHistory()

## Роли
| Роль | Lead | Участники |
|---|---|---|
| PO | Кузнецова Д. | Кузнецова Д., Морозов Е. |
| Аналитик | Волков Г. | Волков Г., Зайцев Ж. |
```

**Многоуровневый Code Review**

| Уровень | Кто | Что проверяет |
|---|---|---|
| 1. Агентный pre-review | AI-агент | Стиль, lint, очевидные баги, покрытие тестами |
| 2. Peer review | Другой разработчик | Бизнес-логика, edge cases |
| 3. Module owner review | Owner модуля | Cross-module влияние, API изменения |
| 4. Архитектурный review | Архитектор | Только для shared code и API контрактов |

### Новые роли при масштабировании

| Роль | Когда появляется | Зачем агент |
|---|---|---|
| Tech Lead | 20+ человек | Агент помогает с архитектурными review, мониторинг техдолга |
| Engineering Manager | 20+ человек | Агент помогает с метриками команды, планированием_capacity |
| DevOps / SRE | 20+ человек | Агент помогает с CI/CD конфигурацией, мониторингом, инцидентами |
