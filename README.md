# Как внедрить AI-агентов в команду разработки

Это практическое руководство для команды цифрового продукта.

Цель: начать использовать AI-агентов так, чтобы команда работала быстрее, но
не потеряла качество, ответственность и понятный процесс.

## Главная мысль

AI-агент не заменяет специалиста.

Агент помогает быстрее делать черновики, искать пропуски, писать код,
готовить тесты и проверять результат. Но решение принимает человек.

Коротко:

```text
человек ставит задачу -> агент помогает выполнить -> человек проверяет и принимает
```

Вся работа должна оставлять след:

- что делали;
- кто владелец;
- какие были границы;
- как проверяли;
- что получилось;
- что делать дальше.

Если это осталось только в чате, процесс нельзя повторить и нормально
проверить.

## Для какой команды

Для команды примерно из 10 человек:

- продакт-оунер;
- архитектор;
- системные аналитики;
- дизайнеры;
- разработчики;
- QA-инженеры.

Эти люди остаются владельцами результата. AI-агенты только помогают им быстрее
готовить и проверять рабочие материалы.

## Что должно появиться в репозитории

Перед массовым использованием агентов нужны общие правила и шаблоны.

Минимальная структура:

```text
project-repo/
├── AGENTS.md
├── README.md
├── docs/
│   ├── process/
│   ├── architecture/
│   ├── product/
│   └── qa/
├── templates/
├── runtime/
│   ├── task-accounting/
│   └── task-artifacts/
└── src/
```

Что за что отвечает:

| Место | Зачем нужно |
| --- | --- |
| `AGENTS.md` | главные правила для AI-агентов |
| `docs/process/` | роли, порядок работы, проверки, правила передачи задач |
| `docs/architecture/` | архитектура, границы модулей, важные решения |
| `docs/product/` | описание продукта, список задач, пользовательские истории |
| `docs/qa/` | тестовая стратегия и правила проверки |
| `templates/` | шаблоны документов |
| `runtime/task-accounting/` | список задач, выбранных в работу |
| `runtime/task-artifacts/` | карточки задач, журналы запусков, результаты проверок |

`runtime/` можно вводить не сразу. Но если команда хочет повторяемый процесс с
агентами, этот слой нужен: он хранит не код, а след выполнения задач.

## Простые термины

В этом руководстве используются несколько рабочих терминов.

| Термин | Простое объяснение |
| --- | --- |
| Артефакт | документ или файл, который остается после работы |
| Карточка задачи | главный документ конкретной задачи: цель, границы, проверки |
| Контракт запуска | короткий план запуска агента: что читать, что делать, что нельзя |
| Журнал запуска | запись того, что агент реально сделал и какие команды запускал |
| Документ проверки | отдельная запись проверки результата |
| Запись закрытия | итог: задача закрыта, остановлена или требует продолжения |
| Правила проекта | архитектура, команды, ограничения и локальные договоренности |
| Способ доказать результат | тесты, сборка, проверка, скриншот или отчет |

Можно использовать английские названия внутри инструментов, например
`execution contract` или `run record`, но в описании процесса лучше писать
простыми словами.

## Рабочие роли конвейера

В команде есть реальные должности: PO, архитектор, аналитик, дизайнер,
разработчик, QA.

Для работы с агентами удобнее отдельно выделить рабочие роли конвейера. Это не
новые должности, а режимы работы.

| Роль конвейера | Что делает |
| --- | --- |
| Менеджер маршрута | выбирает одну задачу, проверяет границы, передает дальше |
| Архитектор | проверяет архитектуру, ограничения и влияние на систему |
| Планировщик | разбивает работу на понятные шаги и готовит карточку задачи |
| Исполнитель | делает изменение: код, документ, тесты или настройки |
| Ревьюер | проверяет результат и фиксирует замечания |

Один человек может временно выполнять несколько таких ролей, но в документах
нужно писать, какая роль сейчас отвечает за шаг.

## Типы проходов

Задачи бывают разными. Поэтому не стоит все вести одним большим промптом.

Основные типы проходов:

| Тип прохода | Когда нужен | Что должно получиться |
| --- | --- | --- |
| Разработка | нужно изменить код, документацию, настройки или процесс | изменение + проверка |
| QA | нужно проверить качество, риски, сценарии или регресс | отчет проверки |
| Декомпозиция | задача слишком большая и ее нужно разбить | несколько маленьких задач |

Если задача смешивает все сразу, ее лучше разделить. Агент хуже работает, когда
его просят одновременно придумать план, написать код, проверить себя и закрыть
задачу.

## Как выглядит один нормальный проход задачи

Нормальный проход выглядит так:

```text
выбрали одну задачу
-> завели карточку задачи
-> собрали контекст
-> составили контракт запуска
-> агент выполнил ограниченный шаг
-> сохранили журнал запуска
-> провели проверку
-> закрыли задачу или записали следующий шаг
```

Что важно:

- задача выбирается из списка работ, а не из случайной фразы в чате;
- на одну карточку задачи не нужно вешать несколько независимых целей;
- агент не должен начинать изменение, если не хватает обязательного контекста;
- результат должен быть проверен человеком;
- итог должен быть записан.

## Как давать задачу агенту

Плохой вариант:

```text
Сделай фичу оплаты.
```

Хороший вариант:

```text
Возьми задачу PAY-42.
Карточка задачи: runtime/task-artifacts/active/PAY-42.md.
Работай только в src/features/payments/.
Прочитай AGENTS.md и docs/architecture/module-boundaries.md.
После изменений запусти тесты payments и git diff --check.
Не меняй auth и shared без отдельного согласования.
Запиши результат проверки в карточку задачи.
```

Задача для агента должна содержать:

- ссылку на карточку задачи;
- границы изменений;
- обязательные документы;
- команды проверки;
- что нельзя менять;
- где записать результат.

## Карточка задачи

Карточка задачи — главный документ конкретной работы.

В ней должно быть:

- ID;
- статус;
- владелец;
- текущая рабочая роль;
- следующая рабочая роль;
- цель;
- что входит в работу;
- что не входит;
- обязательный контекст;
- рабочий контекст;
- критерии готовности;
- план проверки;
- команды;
- результаты проверок;
- вопросы и риски;
- следующий шаг.

Пример:

```markdown
# PAY-42: Оплата банковской картой

## Статус
В работе

## Владелец
Системный аналитик

## Текущая роль
Планировщик

## Следующая роль
Исполнитель

## Цель
Пользователь может оплатить заказ банковской картой.

## Входит в работу
- форма оплаты;
- валидация карты;
- обработка успешной оплаты;
- обработка ошибки оплаты.

## Не входит в работу
- сохранение карты;
- рассрочка;
- возвраты.

## Обязательный контекст
- AGENTS.md
- docs/architecture/module-boundaries.md

## Критерии готовности
- пользователь видит понятную ошибку при отказе платежа;
- успешная оплата переводит заказ в статус `paid`;
- API-контракт согласован с backend.

## Проверка
- unit-тесты;
- интеграционный тест оплаты;
- ручная проверка успешной оплаты и ошибки.
```

## Контракт запуска

Перед запуском агента полезно записать короткий контракт запуска.

Он отвечает на вопросы:

- что агент должен прочитать перед работой;
- какую роль он сейчас выполняет;
- какой маленький шаг он делает;
- что ему можно менять;
- что менять нельзя;
- какие команды нужно запустить;
- куда записать результат.

Контракт запуска нужен, чтобы агент не начинал работу по памяти или по
догадкам.

## Журнал запуска

После работы агента нужен журнал запуска.

В нем фиксируется:

- что агент сделал;
- какие файлы изменил;
- какие команды запускал;
- какие проверки прошли;
- какие проверки не запускались и почему;
- какие риски остались.

Если команда не хранит журнал запуска, проверка превращается в разбор слов из
чата. Это плохо масштабируется.

## Проверка и закрытие

AI-агент может сделать первый проход проверки.

Он хорошо подходит для:

- поиска очевидных ошибок;
- проверки стиля;
- поиска пропущенных тестов;
- проверки структуры документа;
- подготовки списка вопросов.

Но финальное решение принимает человек.

| Проверка | Кто отвечает |
| --- | --- |
| Соответствие задаче | владелец задачи |
| Архитектура | архитектор или tech lead |
| Код | ревьюер |
| Тесты и риски | QA |
| Готовность к релизу | PO + команда |

После проверки нужно записать итог:

- задача закрыта;
- задача закрыта со следующим шагом;
- задача остановлена;
- задача требует разделения;
- задача требует решения человека.

## Обязательный контекст

Агенту нельзя давать задачу в пустоту.

Перед работой нужно разделить контекст на три группы.

| Тип контекста | Что это |
| --- | --- |
| Обязательный | правила, которые нельзя нарушать |
| Рабочий | файлы и документы для этой задачи |
| Дополнительный | внешние статьи, документация, примеры |

Пример:

```text
Обязательный контекст:
- AGENTS.md
- docs/architecture/module-boundaries.md

Рабочий контекст:
- runtime/task-artifacts/active/PAY-42.md
- src/features/payments/
- templates/api-specification.md

Дополнительный контекст:
- документация платежного провайдера
```

Если обязательного контекста нет, агент должен остановиться и попросить
уточнение. Он не должен додумывать правила.

## Проверки

Каждая задача должна иметь проверку.

Для кода это могут быть:

- unit-тесты;
- интеграционные тесты;
- линтер;
- сборка;
- ручная проверка в браузере или приложении.

Для документов это могут быть:

- проверка структуры;
- проверка владельца роли;
- `git diff --check`;
- проверка ссылок;
- согласование с архитектурой или PRD.

Важно: если задача только про документацию, это не значит, что проверка не
нужна. Просто проверка будет документной.

## Шаблоны

В проекте лучше сразу завести папку `templates/`.

Минимальный набор:

```text
templates/
├── task-card.md
├── launch-contract.md
├── run-log.md
├── review.md
├── closeout.md
├── prd.md
├── user-story.md
├── task-specification.md
├── api-specification.md
├── ui-specification.md
├── test-plan.md
├── bug-report.md
└── pr-description.md
```

Первые пять шаблонов нужны для самого процесса работы с агентами. Остальные —
для продуктовых, аналитических, дизайнерских, разработческих и QA-артефактов.

## Навыки агентов

Навык агента — это узкая инструкция для часто повторяющейся работы.

Например:

- написать API-контракт;
- проверить PR на очевидные ошибки;
- подготовить тест-кейсы;
- обновить заметки к релизу;
- проверить структуру документа.

Каждый навык должен отвечать:

- когда его использовать;
- что нужно на входе;
- что должно получиться;
- как проверить результат;
- когда нужен человек.

Плохой навык:

```text
developer-helper
```

Слишком широко. Непонятно, когда запускать и чего ждать.

Хороший навык:

```text
write-api-contract
```

Понятно, что навык помогает написать API-контракт, а не делает всю задачу
целиком.

Навыки можно хранить в репозитории или собирать в формат конкретного
AI-инструмента. Но исходные правила процесса должны жить в обычных документах
проекта, а не только внутри экспортированных навыков.

## Как начать пилот

Не надо начинать с полной автоматизации.

Пилот лучше запускать так:

1. Создать `AGENTS.md`.
2. Создать 3-5 шаблонов: карточка задачи, контракт запуска, журнал запуска,
   проверка, закрытие.
3. Выбрать 3-5 небольших реальных задач.
4. Для каждой задачи завести карточку.
5. Дать агенту только один ограниченный шаг.
6. Сохранить журнал запуска и проверки.
7. Провести проверку человеком.
8. Обновить правила по итогам ошибок.

Цель пилота — не показать магию, а получить повторяемый способ работы.

## План внедрения

| Фаза | Что делаем | Готово, когда |
| --- | --- | --- |
| 1. Подготовка | `AGENTS.md`, роли, первые шаблоны | команда знает, где лежат правила |
| 2. Первые задачи | проводим 3-5 задач через карточки и проверки | есть реальные журналы запусков |
| 3. Ролевые навыки | оформляем частые действия как навыки | у каждой роли есть 1-2 полезных навыка |
| 4. Проверки | закрепляем тесты, проверку и закрытие | результат можно проверить без чата |
| 5. Улучшение | измеряем скорость, качество и переделки | скорость растет без роста хаоса |

## Метрики

Нужно измерять не только скорость.

| Категория | Что смотреть |
| --- | --- |
| Скорость | время от идеи до результата, время работы над задачей, количество завершенных задач |
| Качество | баги, возвраты с проверки, пропущенные требования |
| Проверка | время проверки, частые замечания, повторные правки |
| Проверяемость | сколько задач имеют карточку, журнал запуска и итог |
| Использование AI | сколько людей реально используют агентов |
| Улучшение процесса | сколько правил и шаблонов обновлено после пилота |

Если команда стала делать больше задач, но выросло количество багов и переделок,
это не успех. Это ускорение плохого процесса.

## Частые ошибки

| Ошибка | Почему плохо | Как правильно |
| --- | --- | --- |
| Давать агенту задачу без контекста | агент угадывает и ошибается | давать ссылки на правила и карточку задачи |
| Хранить решения только в чате | знания теряются | переносить важное в документы |
| Верить агенту без проверки | можно пропустить баги | человек всегда проверяет результат |
| Делать слишком широкие навыки | агент выходит за границы | делать узкие навыки |
| Не писать критерии готовности | непонятно, когда задача закончена | фиксировать критерии в карточке |
| Не запускать проверки для задач по документации | документы тоже могут ломать процесс | проверять документ и запускать `git diff --check` |
| Менять чужие модули без согласования | появляются конфликты | соблюдать границы владения |
| Смешивать план, код и QA в одном шаге | результат трудно проверить | разделять задачу на проходы |
| Не записывать закрытие | следующие шаги теряются | фиксировать итог и следующий шаг |

## Масштабирование

При росте команды нужно добавлять не больше хаоса, а больше ясности.

| Размер команды | Что добавить |
| --- | --- |
| До 10 человек | общие правила, шаблоны, базовые навыки |
| 20+ человек | владельцы модулей, role leads, карта проекта |
| 50+ человек | правила для команд, гильдии, релизные проверки |

При росте особенно важны:

- владельцы модулей;
- единый `AGENTS.md`;
- карта проекта;
- правила изменения общих компонентов;
- обязательная проверка человеком;
- понятное закрытие задач.

## Короткий чеклист старта

- [ ] Создать `AGENTS.md`.
- [ ] Описать рабочие роли конвейера.
- [ ] Создать папку `templates/`.
- [ ] Завести шаблоны карточки задачи, контракта запуска, журнала запуска,
      проверки и закрытия.
- [ ] Описать архитектурные границы.
- [ ] Договориться о командах проверки.
- [ ] Выбрать первые 3-5 задач для пилота.
- [ ] После каждой задачи сохранять журнал запуска и результат проверки.
- [ ] Разобрать ошибки пилота и обновить правила.
