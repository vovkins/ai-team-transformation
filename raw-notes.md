## 2026-04-19

### Мысль 1: Набор скиллов по умолчанию для каждой роли

Для каждой роли (аналитика, разработчика, дизайнера, QA, продакт-оунера) надо предложить набор скиллов по умолчанию, который поможет им делать работу более эффективно.

Примеры:
- **PO:** скилл для брейншторма идей user stories + скилл для формулирования user stories в AI-friendly формате
- (остальные роли — TBD)

### Мысль 2: Формат артефактов — AI-friendly + human-readable

Нужно продумать форматы артефактов, которые создаются и используются разными ролями в проекте. Формат должен быть:
- **AI-friendly** — агент может парсить, понимать, заполнять
- **Human-readable** — людям понятно и удобно работать

Это два potentially conflicting требования — нужно найти баланс. Возможно, структурированный markdown с чёткими секциями и обязательными полями.

### Мысль 3: Стандарты для артефактов

Артефакты должны использовать общепринятые стандарты для своего типа:
- Спецификации API — OpenAPI/Swagger
- Технические спецификации — OpenSpec или аналог
- И т.д.

Это облегчит взаимодействие между ролями и агентами — каждый знает, чего ожидать.

### Мысль 4: Готовые роли из существующих репозиториев

Существуют готовые репозитории с ролями для различных фреймворков (включая Codex). Это готовые роли в общепризнанном формате. Нужно подумать:
- Какие готовые роли можно сразу подтянуть в проект
- Как снабдить профильных специалистов нужным набором инструментов из коробки
- Не нужно ли создавать свои роли, если готовых не хватает

Преимущество: не изобретать велосипед, а переиспользовать проверенные решения.

### Мысль 5: Добавить роль Архитектора

В список ролей проекта нужно добавить Архитектора. До того как аналитик и разработчик начнут работать над бэклогом, необходимо:
- Сформулировать общую архитектуру проекта
- Определить архитектурные стандарты и гайдлайны

Архитектор закладывает фундамент, на котором строится вся остальная работа.
