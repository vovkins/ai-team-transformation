# Workflow-примеры по ролям

Этот документ показывает пошагово, как каждая роль работает с AI-агентом на протяжении всего цикла разработки (SDLC). Для каждого этапа: что делает человек, что — агент, какой skill использовать, какой промпт дать и какой артефакт получить на выходе. Примеры построены вокруг одного сквозного проекта — инвестиционного мобильного приложения.

---

## 1. PO (Продакт-оунер)

### Этап 1. Продуктовое исследование

- **Задача:** Изучить рынок, конкурентов, сформулировать гипотезы
- **Skill:** `competitive-analysis`
- **Пример промпта:**

```
Исследуй рынок мобильных банковских приложений в России.
Основные конкуренты: Тинькофф, Альфа, ВТБ.
Нас интересует: UX онбординга, фичи портфеля, подписочные модели.
Результат — Product Research Report по шаблону templates/product-research.md
```

- **Разделение труда:** Агент собирает данные, структурирует. PO оценивает релевантность, формулирует гипотезы.
- **Артефакт:** Product Research Report

### Этап 2. Формирование vision (PRD)

- **Skill:** `jobs-to-be-done`
- **Пример промпта:**

```
На основе Product Research Report (docs/product-research.md)
помоги сформулировать PRD для инвестиционного мобильного приложения.
Целевая аудитория: начинающие инвесторы 25-40 лет.
Ключевое отличие: упрощённый UX + AI-рекомендации.
Результат — PRD по шаблону templates/prd.md
```

- **Разделение труда:** Агент структурирует, дополняет разделы. PO формулирует vision, определяет MVP.
- **Артефакт:** PRD → передаётся Архитектору и Дизайнеру

### Этап 3. Декомпозиция в User Stories

- **Skills:** `brainstorm-stories` + `write-user-story`
- **Пример промпта:**

```
На основе PRD (docs/prd.md) декомпозируй продукт на user stories.
Используй формат INVEST, acceptance criteria в BDD (Given/When/Then).
Группируй по эпикам: Онбординг, Портфель, Торговля, Аналитика.
Результат — backlog по шаблону templates/user-story.md
```

- **Разделение труда:** Агент генерирует черновики. PO проверяет, корректирует приоритеты.
- **Артефакт:** Product Backlog

### Этап 4. Приоритизация бэклога

- **Skill:** `agile-product-owner`
- **Пример промпта:**

```
Помоги приоритизировать backlog (docs/backlog.md).
Учитывай: бизнес-ценность, техническую сложность, зависимости.
Предложи порядок реализации для MVP.
```

- **Разделение труда:** Агент анализирует зависимости. PO принимает финальное решение.
- **Артефакт:** Приоритизированный Backlog

### Итог workflow PO

```
Product Research → PRD → Backlog → Приоритизированный Backlog
      ↓               ↓         ↓            ↓
   (PO+agent)    (PO+agent) (PO+agent)  (PO+agent)
```

## 2. Архитектор

### Этап 1. Проектирование архитектуры

- **Вход:** PRD, Product Backlog
- **Skill:** `define-architecture`
- **Пример промпта:**

```
На основе PRD (docs/prd.md) спроектируй архитектуру инвестиционного приложения.
Стек: React Native (mobile), Node.js + TypeScript (backend), PostgreSQL.
Учитывай: feature-based модульность, строгую типизацию.
Результат — Architecture Document по шаблону templates/architecture-document.md
```

- **Разделение труда:** Агент предлагает паттерны, оценивает стек. Архитектор принимает решения, обосновывает выбор.
- **Артефакт:** Architecture Document

### Этап 2. Формирование стандартов (CONVENTIONS.md)

- **Skill:** `define-architecture`
- **Пример промпта:**

```
На основе Architecture Document сформируй CONVENTIONS.md:
- Правила naming и форматирования
- Архитектурные паттерны (что использовать, что запрещено)
- Обязательные практики (typing, testing, error handling)
Результат — CONVENTIONS.md в корне проекта
```

- **Разделение труда:** Агент генерирует черновик. Архитектор согласовывает с командой.
- **Артефакт:** CONVENTIONS.md

### Этап 3. Определение модульных границ

- **Skill:** `define-architecture`
- **Пример промпта:**

```
Раздели проект на модули на основе Architecture Document.
Для каждого модуля определи: назначение, ключевые файлы, зависимости от других модулей.
Результат — Module Boundaries по шаблону templates/module-boundaries.md
```

- **Разделение труда:** Агент анализирует зависимости и предлагает структуру. Архитектор валидирует границы.
- **Артефакт:** Module Boundaries

### Этап 4. Архитектурный review (непрерывно)

- **Skill:** `review-architecture`
- **Пример промпта:**

```
Проверь PR #{number} на соответствие архитектурным стандартам:
- Не нарушает ли модульные границы
- Соответствует ли CONVENTIONS.md
- Нет ли запрещённых паттернов
Результат — Architecture Review по шаблону templates/architecture-review.md
```

- **Разделение труда:** Агент выявляет нарушения. Архитектор оценивает серьёзность и принимает решение (fix / accept / discuss).
- **Артефакт:** Architecture Review

### Итог workflow Архитектора

```
PRD + Backlog → Architecture Document → CONVENTIONS.md → Module Boundaries
                        ↓                                          ↓
               (Архитектор+agent)                          (Архитектор+agent)
                                                                    ↓
                                              PR Review ← (непрерывно, Архитектор+agent)
```

## 3. UI/UX Дизайнер

### Этап 1. Проектирование дизайн-системы

- **Вход:** PRD, бренд-гайды
- **Skill:** `figma-design-system`
- **Пример промпта:**

```
На основе PRD создай структуру дизайн-системы:
- Design Tokens (цвета, типографика, spacing)
- Базовые компоненты (кнопки, инпуты, карточки)
- Принципы accessible design
Результат — Design System по шаблону templates/design-token.md
```

- **Разделение труда:** Агент генерирует токены и структуру компонентов. Дизайнер оценивает визуальную согласованность, корректирует palette/typography.
- **Артефакт:** Design System

### Этап 2. Проектирование UI Specs

- **Skill:** `figma-generate-design`
- **Пример промпта:**

```
На основе PRD и Design System спроектируй экраны для фичи "Онбординг":
- Все состояния: empty, loading, error, success
- Interaction patterns и переходы
Результат — UI Specifications по шаблону templates/ui-specification.md
```

- **Разделение труда:** Агент описывает экраны и состояния. Дизайнер финализирует layout, micro-interactions, визуальную иерархию.
- **Артефакт:** UI Specifications

### Этап 3. Конвертация в код

- **Skill:** `figma-implement-design`
- **Пример промпта:**

```
Конвертируй UI Spec для экрана Onboarding в React Native компоненты.
Используй Design Tokens из DESIGN-SYSTEM.md.
Соблюдай CONVENTIONS.md.
```

- **Разделение труда:** Агент генерирует код. Дизайнер проверяет соответствие макетам.
- **Артефакт:** UI-компоненты в репозитории

### Итог workflow Дизайнера

```
PRD + бренд-гайды → Design System → UI Specs → UI-компоненты
                        ↓               ↓            ↓
                  (Дизайнер+agent) (Дизайнер+agent) (Дизайнер+agent)
```

## 4. Системный аналитик

### Этап 1. Декомпозиция в технические спецификации

- **Вход:** PRD, User Stories, UI Specs, Architecture Document
- **Skill:** `write-task-spec`
- **Пример промпта:**

```
На основе US-42 ("Регистрация пользователя") создай Task Specification:
- Functional и Non-Functional требования
- API контракты (endpoints, request/response)
- Data model (Mermaid)
- Edge cases
Результат — TS-42 по шаблону templates/task-specification.md
```

- **Разделение труда:** Агент генерирует структуру TS и заполняет разделы. Аналитик проверяет полноту, уточняет edge cases, согласовывает с командой.
- **Артефакт:** Task Specification (TS)

### Этап 2. Валидация спецификаций

- **Skill:** `validate-against-architecture`
- **Пример промпта:**

```
Проверь TS-42 на соответствие:
- Architecture Document (модуль auth/)
- CONVENTIONS.md (стиль, паттерны)
- API контракты других модулей (нет ли конфликтов)
Выдели несоответствия и предложи исправления.
```

- **Разделение труда:** Агент выявляет несоответствия. Аналитик оценивает и вносит правки.
- **Артефакт:** Валидированная Task Specification

### Этап 3. Описание API контрактов

- **Skill:** `write-task-spec`
- **Пример промпта:**

```
На основе Task Specification TS-42 создай API Specification в формате OpenAPI 3.x:
- Endpoint: POST /api/auth/register
- Request/Response схемы
- Error codes
- Примеры
```

- **Разделение труда:** Агент генерирует OpenAPI-спеку. Аналитик проверяет согласованность с другими API, согласовывает с бэкенд-разработчиками.
- **Артефакт:** API Specification (OpenAPI)

### Итог workflow Системного аналитика

```
PRD + US + UI Specs + Arch Doc → Task Spec → Валидация → API Spec
                                       ↓           ↓           ↓
                                 (Аналитик+agent) (Аналитик+agent) (Аналитик+agent)
```

## 5. Разработчик

### Этап 1. Реализация по спецификации

- **Вход:** Task Specification, UI Spec, API Spec
- **Skill:** `implement-feature` (или `frontend-skill`)
- **Пример промпта:**

```
Реализуй TS-42: Регистрация пользователя.
Границы: только src/features/auth/
Критерии: тесты проходят, покрытие ≥80%, соответствует CONVENTIONS.md
Не трогай: shared/, другие модули, auth middleware
```

- **Разделение труда:** Агент пишет код в рамках заданных границ. Разработчик проверяет логику, интегрирует, фиксит edge cases.
- **Артефакт:** Реализованная фича + PR

### Этап 2. Написание тестов

- **Пример промпта:**

```
На основе TS-42 напиши unit tests:
- Покрытие всех acceptance criteria
- Edge cases из спецификации
- Minimum coverage: 80%
Фреймворк: Jest + React Native Testing Library
```

- **Разделение труда:** Агент генерирует тесты. Разработчик проверяет релевантность, добавляет missed cases.
- **Артефакт:** Unit-тесты

### Этап 3. Code review

- **Skill:** `code-review`
- **Пример промпта:**

```
Сделай автоматический code review PR #{number}:
- Соответствие CONVENTIONS.md
- Покрытие тестами
- Очевидные баги
- Security issues
Результат — Review Comments
```

- **Разделение труда:** Агент находит проблемы. Разработчик оценивает замечания, принимает или отклоняет.
- **Артефакт:** Review Comments (автоматические)

### Итог workflow Разработчика

```
TS + UI Spec + API Spec → Код → Тесты → PR → Code Review
                              ↓       ↓              ↓
                        (Разработчик+agent) (Разработчик+agent) (Разработчик+agent)
```

## 6. QA-инженер

### Этап 1. Проектирование тест-стратегии

- **Вход:** Task Specification, API Spec
- **Skill:** `generate-tests`
- **Пример промпта:**

```
На основе TS-42 сгенерируй тест-кейсы:
- Формат: Gherkin (Given/When/Then)
- Покрытие всех acceptance criteria
- Edge cases из спецификации
- Негативные сценарии
Результат — Test Plan по шаблону templates/test-plan.md
```

- **Разделение труда:** Агент генерирует тест-кейсы. QA проверяет полноту покрытия, корректирует приоритеты.
- **Артефакт:** Test Plan

### Этап 2. Написание автотестов

- **Skill:** `playwright`
- **Пример промпта:**

```
На основе Test Plan для TS-42 создай E2E тесты:
- Инструмент: Playwright
- Сценарии: happy path + error cases
- Интеграция с CI
```

- **Разделение труда:** Агент пишет E2E-тесты. QA проверяет стабильность, добавляет flaky-test handling.
- **Артефакт:** E2E-тесты

### Этап 3. Анализ покрытия

- **Skill:** `analyze-coverage`
- **Пример промпта:**

```
Проанализируй покрытие тестами фичи "Регистрация":
- Какие acceptance criteria покрыты
- Какие edge cases не покрыты
- Рекомендации по дополнительным тестам
```

- **Разделение труда:** Агент выявляет gaps. QA приоритизирует дополнения, инициирует доработку.
- **Артефакт:** Отчёт о покрытии + рекомендации

### Итог workflow QA

```
TS + API Spec → Test Plan → E2E тесты → Анализ покрытия
                    ↓            ↓              ↓
              (QA+agent)   (QA+agent)     (QA+agent)
```

## Общая схема взаимодействия ролей

```
PO                Архитектор          Дизайнер          Аналитик         Разработчик        QA
  │                   │                  │                  │                 │               │
  ├─ Research ──┐     │                  │                  │                 │               │
  ├─ PRD ───────┼─────┼──────────────────┼──────┐           │                 │               │
  ├─ Backlog    │     ├─ Arch Doc ───────┤      │           │                 │               │
  └─ Priority   │     ├─ CONVENTIONS     │      │           │                 │               │
                │     ├─ Modules         │      │           │                 │               │
                │     │                  ├─ DS   │           │                 │               │
                │     │                  ├─ UI   │           │                 │               │
                │     │                  │  Specs│           │                 │               │
                │     │                  │      ├─ TS ──────┤                 │               │
                │     │                  │      │  API Spec ├─ Implementation ├─ Test Plan   │
                │     │                  │      │           │  + Unit Tests   ├─ E2E Tests    │
                │     ├─ Arch Review ←───┤      │           │  + PR           ├─ Coverage     │
                │     │                  │      │           │  Code Review ←──┤               │
                ▼       ▼                  ▼      ▼           ▼                 ▼               ▼
```

Каждая стрелка — это артефакт, который передаётся downstream. AI-агент участвует на каждом этапе, ускоряя рутину и повышая качество. Человек принимает решения и несёт ответственность за результат.
